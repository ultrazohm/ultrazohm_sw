# PWM_and_SS_control_V4 IP Core Versions

This folder contains all IP core versions generated by HDL Coder from the Simulink model
`IP_Core_SS_Switch_and_PWM/PWM_and_Switching_Signal_Control`.

Each subfolder (e.g. `PWM_and_SS_control_V4_ip_v4_4`) is a self-contained Vivado IP package.

## Known Issue: Vendor field depends on the machine running HDL Coder

HDL Coder does not expose a setting for the IP vendor field. When Vivado packages the IP via
`vivado_ip_package.tcl`, the vendor is taken from the local Vivado user settings ("Organization"
field in the IP packager). This is machine-dependent, e.g. 

- **Linux/Ubuntu**: defaults to `user.org`
- **Windows**: defaults to `domainname` or `dnsdomainname` or `xilinx.com`
- see also https://www.mathworks.com/matlabcentral/answers/410510-how-can-i-set-the-vendor-for-an-ip-core-generated-by-hdl-coder

This means the VLNV of a generated IP differs depending on who runs HDL Coder, breaking
`upgrade_ip` because Vivado requires the vendor to match across all versions in an upgrade chain.

Versions v4.3 were generated on Windows and deployed with `xilinx.com` as the vendor (no
explicit vendor line in their TCL scripts, so Vivado used the Windows default). Version v4.4 has
been explicitly patched to match:

```tcl
set_property vendor {xilinx.com} [ipx::current_core]
```

This is required so that `upgrade_ip` in Vivado can find the upgrade path from v4.3 → v4.4
(vendor must match across versions in the upgrade chain).

## What to do when generating a new IP version

After running `hdlworkflow.m` in MATLAB, a new subfolder will be created (e.g.
`PWM_and_SS_control_V4_ip_v4_5`). Before committing, manually add the vendor line to its
`prj_ip/vivado_ip_package.tcl`:

1. Open `PWM_and_SS_control_V4_ip_vX_Y/prj_ip/vivado_ip_package.tcl`
2. Find the `# Set IP properties` block:

   ```tcl
   set_property version {X.Y} [ipx::current_core]
   set_property library {ip} [ipx::current_core]
   ```

3. Add the vendor line directly after `set_property library`:

   ```tcl
   set_property vendor {xilinx.com} [ipx::current_core]
   ```

   Use `xilinx.com` to stay consistent with previously deployed versions (v4.4).

4. Run the Vivado packaging script (see next section).

## Packaging the IP in Vivado

After patching `vivado_ip_package.tcl`, run it in Vivado to produce the packaged IP core
(i.e. the `component.xml` and associated files that Vivado reads from the IP catalog).

**Option A — Vivado GUI:**

1. Open Vivado
2. Open the TCL console (**Tools → Tcl Console**)
3. Navigate to the version folder and source the script:

   ```tcl
   cd <repo>/ip_cores/PWM_and_SS_control_ip_v4_0/Simulation/hdl_prj/ipcore/PWM_and_SS_control_V4_ip_vX_Y/prj_ip
   source vivado_ip_package.tcl
   ```

**Option B — Vivado batch mode (command line):**

```bash
vivado -mode batch -source vivado_ip_package.tcl
```

Run this from inside the `prj_ip` folder of the version you want to package.

After the script completes, the Vivado project simply closes without feedback, this is normal.
The `component.xml` in the parent folder is updated and the IP is
ready to be added to the Vivado IP catalog.

## Checking upgrade availability in Vivado

In the Vivado TCL console:

```tcl
# Rebuild the IP catalog after adding/updating IP repo paths
update_ip_catalog -rebuild

# Check which IPs have an upgrade available
get_property UPGRADE_VERSIONS [get_ips *PWM_and_SS_control*]

# Perform the upgrade
upgrade_ip [get_ips *PWM_and_SS_control*]
```
