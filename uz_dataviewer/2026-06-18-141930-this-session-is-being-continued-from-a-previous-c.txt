  Findings

  Bug 1 — JSON restore never recomputes FFT/Histogram (this is your "FFT window 
  not resumed")

  apply_dict restores the sources and all options but never asks the windows to
  recompute:

  # session.py:184  — sets sources, but not compute_requested
  state.fft.sources = [r for r in (_resolve(s) for s in fft.get("sources", []))
  if r is not None]

  The analysis panel only ever computes when compute_requested is set
  (analysis.py:195). So after Load state, the FFT/Histogram windows come back
  with the right source list and options but draw nothing until you manually
  click Compute. The "⚠ stale" hint doesn't even show, because it's gated on
  there already being results (analysis.py:205).

  Notably the script path doesn't have this bug: fft_source/hist_source set
  compute_requested = True (commands.py:567,582), so a replayed .uzscript does
  recompute. The asymmetry is exactly why JSON-restore looks broken and
  script-replay doesn't.

  Fix: in apply_dict, after restoring sources, state.fft.compute_requested = 
  bool(state.fft.sources) (and same for histogram). One line each.

  Bug 2 — Web build can't save/load/export/run sessions at all

  The Session menu is gated on native-only file dialogs:

  # app.py:117
  disabled = pfd is None or bool(self._session_dialog)

  On web pfd is None, so all four items (Save state, Load state, Export script,
  Run script) are permanently greyed out. The data exports (plots/FFT/histogram)
  were given a web path via webbridge.download + the hidden <input type=file>,
  but the Session menu never was.

  Fix, two halves:
  - Save state / Export script (easy): write to /tmp/<name> then
  webbridge.download(path, name), same pattern the data exports already use.
  download() reads UTF-8 text, so JSON/.uzscript work as-is (its hardcoded
  type="text/csv" is cosmetic).
  - Load state / Run script (more work): needs an upload route. The hidden
  uzFileInput currently hardcodes uploaded files to request_load (i.e. treat as
  data). We'd add a "pending upload action" so a .json routes to load_state and
  .uzscript to run_script — a small change in webbridge.py plus the JS change
  handler in build/gen_web.py.
  - Caveat to document: restoring a JSON session re-loads its data files by path
  (load_sync(run.path)), and those paths won't exist in a fresh Pyodide FS — so
  web load-state realistically works within one session or needs the data
  re-uploaded. Out of scope to fully solve here.

  Bug 3 — .uzscript export is lossy vs JSON for analysis options

  to_script (session.py:310-317) emits only fft_source(...) / hist_source(...).
  It drops everything else the JSON saves: FFT remove_dc, Hann window,
  log_x/log_y, follow_plot/range; histogram bins, follow/range; and per-cell
  export_relative. So a script "reproduces the session" with DC-removal/Hann
  silently back on, bins back to 50, and any custom analysis window lost — a
  different picture than the JSON of the same session. The module docstring
  claims both formats reproduce the session.

  Fix: emit the non-default analysis options in to_script
  (fft_remove_dc/fft_hann/fft_logx/fft_logy, fft_follow+fft_range when not the
  default, hist_bins, set_export_relative), guarded on non-default so scripts
  stay short. Commands already exist for all of these.

  Also worth noting

  - No test covers FFT/Histogram round-trip — that's why Bug 1 and Bug 3 slipped
  through. test_session.py checks grid/cells/xy only.

  Proposed plan

  1. Bug 1 (small, highest value): set compute_requested for fft & histogram in
  apply_dict. Add a test_session assertion that a restored session with FFT/hist
  sources flags a recompute.
  2. Bug 3 (correctness): make to_script emit non-default analysis +
  export_relative options; extend the script round-trip test.
  3. Bug 2 (web parity): enable Save state / Export script on web via /tmp +
  download; add the upload route for Load state / Run script in webbridge.py +
  build/gen_web.py. Bigger and touches the JS shim — I'd do it as a separate
  commit.

  I'd suggest doing 1 and 2 together (pure-Python, fully testable now) and
  2-the-web-part as a follow-up. Want me to implement 1 + 3 now, or all three?

● Great — that surfaced some hard numbers that reshape the answer. Two of my
  assumptions were wrong, in useful ways:

  - numpy.fft already preserves float32 → complex64 (modern pocketfft backend).
  Our compute_fft throws that away with .astype(np.float64).
  - Prime/awkward window lengths are catastrophic: rfft of a ~2M prime length
  took 571 ms vs 66 ms padded to the next fast length — ~9×.
  - The numpy "prep" around the FFT (astype + hanning + demean + multiply) costs
  as much as the transform itself (278 ms vs 240 ms at 9M).

  Here's how I'd think about the options, ordered by leverage-per-risk. The
  governing constraint throughout: anything that must run in Pyodide/wasm32
  can't be a native binary lib, and adding a native-only fast path means
  maintaining two implementations (the thing this project deliberately avoided
  when it dropped tsdownsample).

  Tier 1 — pure-NumPy, dual-platform, high leverage (no new deps)

  1. Stop upcasting the FFT to float64. Drop ys.astype(np.float64); keep float32
  → complex64. Halves every large allocation (the 75M-sample full-window FFT
  goes from ~600 MB complex128 to ~300 MB complex64 — also eases the 4 GB WASM
  ceiling) and is faster. Precision is irrelevant for an amplitude spectrum
  you're plotting.

  2. Pad the FFT to next_fast_len. This is the single biggest FFT win and it's
  free (numpy has the machinery; scipy exposes next_fast_len directly, or
  compute the next 2^a·3^b·5^c). ~9× on bad lengths, near-zero on already-smooth
  ones. Caveat: padding interpolates the spectrum (finer bin spacing) and you
  must keep the 2/n amplitude normalization tied to the original n, not the
  padded length — a real correctness detail, and it interacts with the "fixed
  bin count" idea already in FFT_FUTURE_IDEAS.md.

  3. In-place prep + cached window. ys -= ys.mean(), ys *= window, and memoize
  np.hanning(n) by length (often reused across recomputes at the same window).
  With scipy.fft(overwrite_x=True) you also avoid the transform's internal copy.
  This roughly halves the non-FFT half of the cost.

  4. Per-frame decimation cache — now the most important one. Since we just
  disabled idle throttling, every frame re-runs decimate_range for every signal
  even when the view hasn't moved (previously capped at 9 fps). Memoize per
  signal on (x_min, x_max rounded, n_out) and reuse the cached arrays when
  nothing changed. This is the natural complement to "always full FPS" and is
  pure Python — it turns a static-but-full-rate screen from "re-decimate
  everything 60×/s" into "decimate once."

  5. Pixel-aware point budget. Cap the envelope target at the plot's pixel width
  (a 1200 px plot can't show more than ~1200–2400 envelope points). At
  max_points = 5000–10000 you're currently decimating and uploading 2–4× more
  vertices than the screen can resolve — wasted in both decimation and draw.
  Clean, pure-Python, helps every path.

  Tier 2 — architecture (native), no new deps

  6. Build the pyramid off the render thread. Pyramid.build is O(n) and
  currently lazy on first render → a visible hitch on a 75M signal. Native
  already has a ThreadPoolExecutor for loading; build the pyramid there. Web
  stays synchronous regardless.

  Tier 3 — new libraries (discuss, but each breaks or strains the single-path 
  design)

  scipy.fft — the only library candidate that's also available in Pyodide. Gives
  workers=-1 (multicore, native only — no-op in web), overwrite_x,
  next_fast_len, and clean single-precision. Modest, real, low-risk if you
  accept scipy as a dependency (it's heavy in the web bundle). Honestly, items
  1–3 capture most of its benefit without the dependency.

  pyFFTW (FFTW) — fastest CPU FFT with plan reuse/wisdom, multithreaded.
  Native-only (no wasm wheel), and its sweet spot is repeated FFTs of the same 
  size — but ours are on-demand over varying windows, so planning overhead
  fights you unless you pad to a small set of fixed lengths (which #2 already
  implies). High complexity for uncertain gain here.

  Numba — JIT the decimation kernels into one fused pass (no intermediates).
  Native-only (LLVM, not in Pyodide) → two implementations, plus first-call
  compile latency. The decimation is already ~0.5 ms/signal, so the headroom is
  small; not worth it.

  JAX — XLA JIT + GPU. I'd specifically not pursue it for this:
  - No jaxlib in Pyodide → breaks web entirely.
  - CPU JAX rarely beats pocketfft for single FFT calls; XLA's advantage is
  fusing pipelines run many times, with per-shape compilation cost — the
  opposite of on-demand, varying-size FFTs.
  - The real JAX/GPU win (cuFFT) only materializes for users with CUDA, on huge
  transforms — a niche for a desktop log viewer, and a heavy, fragile
  dependency.
  - It could only ever make sense as an optional native GPU "turbo" mode, gated
  and fully duplicated against the numpy path. Large effort, narrow payoff.

  My recommendation

  Profile the 75M file first to confirm where the on-demand FFT time actually
  goes, then do 1–3 (FFT) and 4–5 (rendering) — all pure-NumPy, both platforms,
  no dependency, and they target exactly what the numbers above show (prep ≈
  transform; prime lengths ≈ 9× cliff; full-FPS re-decimation waste). Reach for
  scipy.fft only if multicore native FFT becomes a stated requirement. Leave
  JAX/FFTW/Numba unless you decide to ship a separate, native-only accelerated
  build — at which point GPU FFT via scipy's CuPy backend or JAX could be a
  dedicated project.
